# AGENTS.md

本文档供在本仓库中工作的自动化编码代理使用。除非用户另有明确要求，所有分析、进度说明和交付总结均使用中文。

## 项目概览

- 本项目是运行在 Android AutoJs6 6.7.0 或兼容环境中的聚合签到脚本，不是 Node.js 应用。
- 主入口为 `main.js`，项目元数据位于 `project.json`。
- `core/` 存放各应用签到实现，公共基类是 `core/BaseSignRunner.js`，新任务可参考 `core/SignRunnerTemplate.js`。
- `lib/` 存放自动化、控件查找、OCR、截图、日志、任务队列等基础能力。
- `config.js`、`config_ex.js`、`signConfig.js` 管理通用配置和签到专用配置。
- `可视化配置.js` 负责启动配置 WebView，前端资源位于 `vue_configs/`，前后端通过 Bridge 通信。
- `extends/` 是用户自定义扩展点。不要把用户本地扩展硬编码进核心配置。
- `test/` 和 `unit/` 中的脚本需要在 AutoJs6 或 Android 设备上运行，不能假定桌面 JavaScript 运行时能够执行。

## 工作原则

- 修改前先阅读相关入口、调用方、配置定义和已有相似实现，不凭文件名猜测行为。
- 保持改动聚焦，避免无关重构。工作区可能包含用户尚未提交的修改，必须保留并绕开这些修改。
- 技术决策优先考虑质量、简单性、健壮性、可扩展性和长期可维护性，不以短期开发成本为主要依据。
- 修复缺陷时，先按终端用户实际操作路径在真实或尽可能接近真实的 E2E 环境中复现，再定位原因和实现修复。
- 发现明显的 UI 瑕疵、测试失败、静态检查问题或测试不稳定时，应一并处理或在无法安全处理时明确说明。
- 不手动修改 `CHANGELOG.md` 或任何注明为自动生成的文件。
- 不在提交信息中自动添加代理名称或 `Co-authored-by`。
- 文档和提交信息中不要使用 em dash 字符，使用普通连字符 `-`。

## Auto.js 兼容性

- 不要把代码自动转换为现代 Node.js、浏览器 ESM 或 TypeScript 写法。
- 保持仓库现有的 CommonJS 模式和 Auto.js 全局 API，例如 `runtime`、`global`、`files`、`engines`、`device`、`app`、`sleep`、`exit`、`ui`。
- 修改语法前确认 AutoJs6 6.7.0 支持该语法。避免无必要地引入构建步骤、第三方包或运行时 polyfill。
- `可视化配置.js` 包含 Auto.js UI DSL，普通 Node.js 解析器可能无法解析，不要将桌面端语法检查结果误判为代码错误。
- 图片、OCR、无障碍节点和坐标逻辑必须考虑不同屏幕尺寸、系统版本、应用版本、加载速度以及权限状态。
- 优先使用稳定的控件属性和已有 `WidgetUtils`、`Automator`、OCR、找图封装。固定坐标只能作为有明确适用范围的后备方案。
- 新增图片配置时沿用 `signConfig.js` 和 `config_data/` 的绑定及存储方式，不要直接提交用户设备生成的敏感数据。

## 代码风格

- 遵循相邻文件的风格。当前核心代码通常使用 2 空格缩进、单引号、CommonJS `require`，且多数语句不写分号。
- 命名应表达业务含义。签到任务及子任务的 `taskCode` 必须稳定且唯一，避免修改已发布标识导致历史执行状态或计划失效。
- 签到实现应继承 `BaseSignRunner`，成功后才调用 `setExecuted()` 或 `setSubTaskExecuted()`，失败路径不得错误标记完成。
- 耗时等待应设置合理上限，并记录足以诊断失败位置的日志。不要用无边界循环掩盖页面状态异常。
- UI 操作前验证目标是否存在及是否可点击，操作后验证页面状态或业务结果，不要仅以点击动作完成作为成功依据。
- 新增配置字段时同步检查默认值、持久化、Bridge、可视化表单、标签和运行时读取路径，避免只改其中一层。
- 注释解释原因、限制或非直观行为，不重复翻译代码本身。
- 日志不得泄露账号、令牌、完整手机号、支付信息或截图中的其他敏感内容。

## 开发流程

1. 用 `git status --short` 确认现有改动，区分用户修改与本次任务修改。
2. 使用 `rg` 查找任务代码、配置键、Bridge 方法和调用链，检查同类任务的成熟实现。
3. 缺陷修复先在 AutoJs6 中按用户路径复现，并记录设备、Android、AutoJs6、目标应用版本、权限和实际日志。
4. 做最小而完整的实现，确保数据库执行状态、调度、资源回收和失败重试语义保持一致。
5. 运行与改动匹配的测试，并人工检查日志、悬浮窗提示及可视化配置页面。
6. 提交前复查 diff，确认没有敏感数据、调试开关、临时截图、设备专用配置或无关格式化。

## 验证要求

仓库没有可替代设备验证的统一桌面端测试命令。验证时根据改动范围执行以下项目：

- 签到逻辑：在 `test/测试签到功能.js` 中临时指向目标 Runner，在受支持的 AutoJs6 环境中运行。测试结束前还原测试入口，不提交临时目标切换。
- 完整流程：从 `可视化配置.js` 启用任务、设置执行时间、生成计划，再通过 `main.js` 或等价调度入口运行，验证执行状态与第二次运行的跳过行为。
- 配置页面：检查首次加载、已有配置回显、修改保存、返回导航、左右滑菜单、不同屏幕宽度以及 WebView Bridge 报错。
- 失败路径：至少验证无障碍未开启、截图权限拒绝、目标控件缺失、网络或页面加载超时等与改动相关的场景。
- 资源管理：检查截图和图像对象回收、悬浮窗关闭、任务队列移除、事件监听清理及异常后的重试行为。
- 静态检查：对不含 Auto.js UI DSL 的普通 JavaScript 文件，可使用可用的解析或检查工具，但不得据此声称完成了设备端验证。

若当前环境没有 Android 设备或 AutoJs6，必须明确列出未执行的 E2E 项目、原因和建议的人工验证步骤，不能将桌面端检查描述为完整通过。

## 新增签到任务

- 先参考 `core/SignRunnerTemplate.js` 和一个结构相近的现有 Runner。
- 在任务注册配置中使用唯一且稳定的 `taskCode`，并正确声明脚本、默认启用状态和子任务。
- 如需额外配置，沿用 `signConfig.js` 的 Binder；用户自定义任务则使用 `extends/CustomConfig.js` 与 `extends/CustomSignConfig.js` 扩展点。
- 若新增或修改可视化设置，同时更新 `vue_configs/js/` 中相关组件、标签、路由或 store，并验证 Bridge 两端字段一致。
- 用独立测试脚本验证成功、失败、重复执行、子任务禁用及恢复运行场景，再验证完整调度链路。

## 交付说明

- 用中文简要说明完成内容、涉及文件和验证结果。
- 对未能在真实设备验证的内容明确标注，不夸大测试覆盖。
- 如发现与任务无关但重要的问题，可在交付中单独列出，避免把未经授权的大范围修改混入当前任务。
